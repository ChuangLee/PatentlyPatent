# patent_king v2 — 企业自助专利挖掘系统 需求文档

> v0.2 草案 · 2026-05-07 · 与 v1（代理师 CLI 全流程）相比，目标用户、形态、检索后端均变更

## 1. 项目目标

为**企业员工**提供"工作中发现创新 → 自助挖掘 → 出交底书"的 **Web 自助平台**，由 LLM agent 引导员工把含糊的工作发现拆成可申请专利的发明点，并通过**智慧芽（PatSnap）开放 API**做现有技术筛查与初步专利性判断，最终输出符合 CN 实务格式的中文交底书供 IP 部门审阅。

## 2. 用户与场景

| 角色 | 主要场景 | 入口 |
|---|---|---|
| **员工**（研发/产品/设计…） | 描述工作中创新 → 跟 agent 一问一答 → 拿到检索报告与交底书初稿 → 提交给 IP 部门 | Web 浏览器 |
| **IP 工程师 / 代理师** | 审阅员工提交的交底书 → 反馈修改 → 立项申请 | Web 浏览器（同站点不同角色） |
| **管理员** | 用户/部门/项目管理、统计、配额 | Web 后台 |

## 3. 形态、部署、技术约束

- **形态**：Web 应用（FastAPI 后端 + 轻量前端）
- **部署**：**企业内网单租户**——交底书内容机密，绝不外发
- **LLM**：通过企业内网 proxy 调用 Anthropic API（或换成内网部署的 Qwen/GLM 等开源模型）
- **专利检索**：唯一后端为**智慧芽开放 API**（凭证已存 `.secrets/zhihuiya.env`）
- **数据存储**：默认 SQLite（小团队），需要时可切 PostgreSQL
- **认证**：账号密码 / 企业 SSO（OIDC/SAML）二选一可插拔

## 4. MVP 范围（v2.0）

**核心 4 阶段闭环**：

| 阶段 | 用户动作 | Agent 动作 | 产出 |
|---|---|---|---|
| ① 报门 | 一句话/多段文字/上传文档 描述工作中创新 | 接收 + 初步识别技术领域 + 决定是否进入引导 | 创新项目卡片 |
| ② 引导拆解 | 跟 agent 问答（问题树） | 5-Why + 1-What-if + 上下位 + 效果矩阵；每轮总结到结构化字段 | 结构化创新摘要（领域/问题/手段/效果/区别于现有） |
| ③ 检索 + 初判 | 一键触发 | 调用智慧芽 API 关键字检索 + 同族过滤 + LLM 对比；对每篇命中文献做"是否破新颖性/创造性"判断 | 检索报告（≤10 篇高相关 + 每篇 X/Y/N 标注） + 初步专利性结论（强/中/弱/不建议） |
| ④ 交底书 | 审阅 + 编辑 + 提交 | 生成 CN 实务标准交底书：技术领域 / 背景 / 现有技术 / 发明内容（多档独权概括度） / 实施例 / 技术效果 | 中文交底书 markdown + 可导出 docx |

## 5. 功能需求

### 5.1 用户与权限
- F-U-1 账号密码登录（最小可用）；预留 OIDC SSO 接入
- F-U-2 三种角色：`employee` / `ip_reviewer` / `admin`
- F-U-3 部门字段；员工只看自己创建的项目，IP reviewer 看本部门，admin 看全部

### 5.2 创新项目（Project）
- F-P-1 创建：标题（10-50 字） + 描述（≥100 字）+ 可选文件上传（pdf/docx/md）
- F-P-2 状态机：`drafting → researching → reporting → submitted → reviewing → approved | rejected`
- F-P-3 时间线：每次 agent 交互、检索、生成都留痕
- F-P-4 项目协作者：可加同部门员工

### 5.3 引导拆解（Mining）
- F-M-1 多轮对话（流式）；每轮 agent 输出"已捕获要素"+ "下一个问题"
- F-M-2 结构化字段实时更新到右侧面板：`领域 / 问题 / 手段 / 效果 / 区别 / 假想对手`
- F-M-3 5-Why、1-What-if、上下位、效果矩阵 4 类引导问句库（人审过的高质量 prompt）
- F-M-4 终止条件：员工点"差不多了"、或 agent 判定信息已够（预设阈值）

### 5.4 智慧芽检索（Search）
- F-S-1 自动从结构化创新摘要生成查询：核心词（中英）+ IPC/CPC（LLM 推荐）
- F-S-2 调用智慧芽 API：检索 + 详情 + 同族 + 引用
- F-S-3 LLM 对每篇命中做"问题/手段/效果"对照 + X/Y/N 标注 + 摘要给员工读
- F-S-4 按项目缓存 API 调用（避免重复消耗配额）
- F-S-5 配额监控：每日/每月调用数；超阈值降级到只检索 top10

### 5.5 初步专利性判断（Patentability）
- F-PT-1 综合检索结果给出 4 档结论：**很可能新颖** / **新颖性边缘** / **创造性存疑** / **建议不申请**
- F-PT-2 用员工能懂的话解释结论（不堆 A22.3 等条款，但保留可展开的"专业版"）

### 5.6 交底书生成（Disclosure）
- F-D-1 模板化生成：技术领域 / 背景技术（基于检索）/ 发明内容（独权 3 档概括度）/ 附图说明（占位）/ 具体实施方式
- F-D-2 流式生成 + 员工实时编辑（左生成 / 右编辑）
- F-D-3 导出 Markdown / Word（.docx，使用 pandoc 或 python-docx）
- F-D-4 提交给 IP reviewer：状态 `submitted`，触发邮件/IM 通知

### 5.7 IP reviewer 审阅
- F-R-1 工作台：本部门待审项目列表
- F-R-2 在线批注 + 状态变更（approved / rejected / 需补）
- F-R-3 反馈通过邮件/系统消息回员工

### 5.8 管理员后台
- F-A-1 用户/部门管理
- F-A-2 智慧芽配额监控、LLM token 用量看板
- F-A-3 项目/审阅统计

## 6. 非功能需求

- N-1 **隐私**：所有交底内容驻留内网；外呼仅限智慧芽 API 与（可选）Anthropic API；可整体切换到内网 LLM
- N-2 **性能**：报门 → 检索 → 交底书全流程 ≤ 8 分钟；单请求响应 < 3s（流式生成除外）
- N-3 **审计**：每次外呼记录请求摘要 + 响应大小（不记 token 明文）
- N-4 **可移植**：SQLite / PostgreSQL 切换无需改代码（SQLAlchemy）
- N-5 **可观测**：结构化日志 + Prometheus metrics 接口

## 7. 边界（agent 不替员工/IP reviewer 决策）

| 谁决策 | 决策点 |
|---|---|
| 员工 | 是否觉得自己的"创新"值得报；交底书最终内容定稿 |
| IP reviewer | 是否立项申请；独权概括度终判；最接近现有技术 D1 重定义 |
| Agent 仅出"候选 + 风险" | 4 档专利性结论、3 档独权草稿、检索 X/Y/N 标注 |

## 8. 与 v1 (`pk` CLI) 的关系

- 复用：`pk/mining/`、`pk/drafting/`、`pk/validity/`、`pk/llm/`、kb 知识库
- 替换：`pk/search/sources/*` 全部废弃 → 新增 `pk/search/zhihuiya.py`（智慧芽客户端）
- 上层新增：`pk/web/`（FastAPI 后端）+ `pk/db/`（SQLAlchemy 模型）+ `pk/auth/`
- v1 的 `pk` CLI 保留，作为 IP 工程师/管理员的命令行工具（批量导出、统计等）

## 9. 验收（MVP v2.0）

1. 员工浏览器登录 → 一句话报门 → 5 轮 agent 引导后得到结构化摘要 → 检索（智慧芽真实调用）≥ 5 篇相关 + X/Y 标注 → 4 档专利性结论 → 交底书草稿可导出 docx — **全流程 ≤ 10 分钟**
2. IP reviewer 看见员工提交的项目，可批注、改状态
3. 智慧芽 API 调用走凭证 + 限速 + 缓存 + 配额监控
4. 至少 3 个真实样例（来自 3 个不同领域：软件 / 机械 / 化学）通过端到端

## 10. 里程碑

- **M1（一周）**：智慧芽客户端 + 数据模型 + 报门→检索 单元跑通（无 web）
- **M2**：FastAPI 后端骨架 + 认证 + 项目 CRUD + 流式 mining
- **M3**：完整 4 阶段 + 极简前端 + 端到端 demo
- **M4**：reviewer 审阅 + 导出 docx + 配额/审计

## 11. 待定 / 风险

- ⚠️ **智慧芽是否提供官方 MCP**：等调研 agent 回报；若有走 MCP，无则走 REST
- ⚠️ **配额**：实测前不知具体阈值；需在缓存层做防御
- ⚠️ **企业 SSO**：MVP 先用账号密码，SSO 留接口
- ⚠️ **附图**：MVP 不生成附图，仅占位；v2.1 接 PatentLMM 类多模态
